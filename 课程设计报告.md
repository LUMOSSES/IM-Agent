# 基于 LangChain4j 的智能对话系统设计与实现

## 摘要

本文设计并实现了一款基于 Spring Boot 3.5 与 LangChain4j 框架的智能对话系统——「小智」。系统以阿里 DashScope (qwen-max) 为核心推理引擎，集成 RAG（检索增强生成）技术实现私有知识库问答，通过 MCP（模型上下文协议）接入外部搜索与时间查询工具，采用 SSE（Server-Sent Events）实现流式对话响应，并提供了完整的会话管理、知识库管理、主题切换、安全护栏等功能。系统采用 B/S 架构，后端基于 Java 17，向量数据库选用 PGVector，会话存储使用 Redis，前端为原生 HTML/CSS/JS 单页应用。

---

## 第一章 绪论

### 1.1 选题背景与意义

2022 年底 ChatGPT 发布以来，大语言模型（LLM）迅速渗透到各行各业。然而通用大模型存在两大痛点：一是知识截止日期限制，无法回答训练数据之后的问题；二是缺乏领域专有知识，在垂直场景下容易产生"幻觉"。

RAG（Retrieval-Augmented Generation）技术通过在推理前检索外部知识库，将相关文档片段注入提示词，有效解决了上述问题。LangChain4j 是 LangChain 的 Java 生态实现，提供了 AI Services、ChatMemory、Tools、RAG 等完整抽象层，极大降低了 Java 开发者构建 AI 应用的复杂度。

本课题面向日常社交场景，构建一个集成私有知识库（社交技巧、心理学、聊天方法）的智能助手，帮助用户获得高质量的社交建议。系统同时支持联网搜索，可处理实时信息查询。

### 1.2 主要工作

1. 基于 LangChain4j AI Services 搭建对话引擎，集成 ChatMemory 实现多轮记忆
2. 基于 PGVector 实现 RAG 检索增强生成，支持来源引用展示
3. 基于 MCP 协议接入外部工具（网络搜索、时间查询）
4. 实现 SSE 流式对话、会话持久化、知识库管理、主题切换等完整前端功能
5. 集成安全护栏与监控指标采集

## 第三章 系统需求分析

### 3.1 功能需求

系统角色分为两类：普通用户和 AI 系统。

**用户功能：**

| 编号 | 功能 | 描述 |
|------|------|------|
| F1 | 流式对话 | 发送消息，AI 逐 token 实时回复，支持中断 |
| F2 | 会话管理 | 新建、切换、删除会话，会话标题自动取首条消息前 14 字 |
| F3 | 会话持久化 | 刷新页面后恢复会话列表和历史消息 |
| F4 | 清空消息 | 一键清空当前会话聊天记录 |
| F5 | 主题切换 | 暗黑/浅色模式一键切换，偏好记忆 |
| F6 | 知识库查看 | 列出所有知识文档及问答数量 |
| F7 | 知识库上传 | 上传 .md 文件，自动向量化导入 |
| F8 | 知识库删除 | 删除文档，含文件名二次确认，软删除至 trash |

**AI 系统功能：**

| 编号 | 功能 | 描述 |
|------|------|------|
| F9 | 知识库检索 | 将用户问题向量化，在 PGVector 中检索 Top5 相似片段（minScore=0.75）|
| F10 | 联网搜索 | 实时信息通过 MCP 调用智谱搜索 |
| F11 | 时间查询 | 通过 @Tool 获取当前上海时间 |
| F12 | 知识自学习 | AI 可将用户提供的信息写入知识库 |
| F13 | 安全护栏 | 拦截含敏感词的输入 |
| F14 | 来源引用 | RAG 回答标注来源文件名 |
| F15 | Token 统计 | 记录每次调用的 Token 消耗与耗时 |

### 3.2 非功能需求

- **实时性**：流式响应首 token 延迟 < 3s
- **可靠性**：消息持久化至 Redis，重启不丢失
- **安全性**：路径穿越防护、敏感词拦截、可选的 JWT 认证
- **响应式**：支持桌面端（双栏）和移动端（单栏）布局

## 第五章 关键实现

### 5.1 AI 服务装配

`AiChatService` 是系统的核心配置类，负责将各组件组装为可用的 AI Service：

```java
AiChat aiChat = AiServices.builder(AiChat.class)
    .chatModel(chatModel)                    // qwen-max 同步模型
    .streamingChatModel(streamingChatModel)  // qwen-max 流式模型
    .contentRetriever(contentRetriever)      // PGVector RAG 检索器
    .chatMemoryProvider(memoryProvider)      // Redis 记忆（最大 20 条）
    .tools(ragTool, timeTool, emailTool)     // 本地 @Tool
    .toolProvider(mcpToolProvider)           // MCP 远程工具
    .build();
```

`AiChat` 接口通过注解声明角色与行为：

```java
@SystemMessage(fromResource = "system-prompt/chat-bot.txt")
Flux<String> streamChat(@MemoryId Long sessionId, @UserMessage String prompt);
```

System Prompt 定义了「小智」的人设：仅依据检索知识回答、实时信息需搜索、引用来标来源、结尾用感叹号等规则。

### 5.2 RAG 检索增强生成

**文档导入流程：**

```
.md 文件 → DocumentByParagraphSplitter(300字/100重叠)
  → textSegmentTransformer(file_name前缀)
  → EmbeddingModel(text-embedding-v4, 1024维)
  → PGVector 向量表
```

**检索流程：**

```
用户提问 → EmbeddingModel 向量化
  → PGVector 余弦相似度搜索
  → Top 5 (minScore = 0.75)
  → 注入 System Prompt 上下文
  → qwen-max 生成回答
```

**来源追踪：**

`SourceTrackingContentRetriever` 装饰标准检索器，通过 ThreadLocal 拦截检索结果，提取文件名、摘要片段（截取前 200 字）、相似度分数。Controller 在流式响应前注入 `__SOURCES__:{json}\n` 前缀，前端解析后渲染为来源引用气泡。同一文件多个命中片段自动去重，仅保留最高分条目。

### 5.3 SSE 流式对话

后端使用 Reactor `Flux.defer()` 确保每次请求独立创建流：

```java
return Flux.defer(() -> {
    Flux<String> chatStream = aiChat.streamChat(sessionId, prompt);
    return Flux.concat(sourcesFlux, chatStream)
        .doOnNext(token -> fullResponse.append(token))
        .doFinally(signal -> appendUiMessage(sessionId, "ai", fullResponse.toString(), sources));
});
```

前端使用 `ReadableStream` 逐块读取，按 `\n` 分割后解析 `data:` 前缀（SSE 协议），实时追加到对话气泡中。非 `__SOURCES__:` 前缀的 token 直接渲染。

### 5.4 MCP 工具集成

MCP（Model Context Protocol）由 Anthropic 提出，用于标准化 AI 模型与外部工具的交互。系统通过 LangChain4j 的 `McpToolProvider` 以 SSE 协议接入智谱 BigModel 搜索服务：

```java
McpToolProvider.builder()
    .streamingChatModel(streamingChatModel)
    .mcpClients(mcpClient)  // BigModel SSE client
    .build();
```

本地 `@Tool`（RagTool、TimeTool、EmailTool）与远程 MCP Tool 在 AiChatService 中统一注册，AI 根据用户意图自动选择调用。

### 5.5 会话管理

- **AI 上下文记忆**：LangChain4j 的 `ChatMemoryProvider` 基于 Redis 存储，保留最近 20 轮对话，TTL 1 小时
- **UI 消息存储**：`StringRedisTemplate` 操作 Redis List，Key 格式 `ui-msgs:{sessionId}`，消息以 JSON 存储（含 role、text、ts、sources），TTL 2 小时
- **前端持久化**：`localStorage` 存储会话列表（sessionId、title、createdAt），刷新后先读 localStorage 再调 `/api/messages` 恢复历史
- **会话标题**：发送首条消息时自动取前 14 个字符作为标题

### 5.6 知识库管理

知识库面板为右侧滑出式设计，支持：

- **文档列表**：扫描 `docs/` 目录下所有 `.md` 文件，统计 `### Q：` 数量作为问答数
- **上传**：FormData multipart 上传，自动向量化导入 PGVector。若文件同名则先清除旧向量再导入
- **查看**：获取文档全文，新窗口展示
- **删除**：软删除至 `docs/trash/` + 调用 `embeddingStore.removeAll(metadataKey("file_name").isEqualTo(name))` 清除向量。前端要求输入完整文件名二次确认

### 5.7 安全机制

- **安全护栏**：`SafeInputGuardrail` 检查用户输入是否包含「死」「杀」等敏感字符，命中则抛出异常由全局异常处理器拦截
- **路径穿越防护**：`KnowledgeController.resolveSafe()` 拒绝包含 `..`、`/`、`\` 的文件名，并校验规范化路径在 `docsPath` 范围内
- **JWT 认证**（集成模式）：基于 jjwt (HS512) 解析 `Authorization` 头，提取 userId。独立模式通过 `agent.auth.enabled: false` 跳过

### 5.8 监控指标

`AiModelMonitorListener` 实现 LangChain4j 的 `ChatModelListener` 接口，拦截每次模型调用的开始、完成、失败事件，记录 userId、sessionId、模型名称、请求/响应 Token 数、耗时。`AiModelMetricsCollector` 将这些数据注册为 Micrometer 指标，暴露至 `/actuator/prometheus` 供 Prometheus 采集。

核心指标：

| 指标名 | 类型 | 标签 |
|--------|------|------|
| `ai_model_requests_total` | Counter | user, session, model, status |
| `ai_model_errors_total` | Counter | user, session, model |
| `ai_model_tokens_total` | Counter | user, session, model, type(input/output/total) |
| `ai_model_response_duration_seconds` | Timer | user, session, model |

### 5.9 前端交互

- **暗黑/浅色主题**：通过 CSS 自定义属性（`data-theme`）切换全局颜色变量。选择持久化至 `localStorage`
- **响应式布局**：CSS Grid 双栏布局（侧边栏 320px + 主区域自适应），`@media (max-width: 768px)` 折叠为单栏
- **动画**：面板滑入使用 `transform: translateX()` + `opacity`（GPU 加速），过渡时间 250ms。支持 `prefers-reduced-motion` 查询
- **Toast 通知**：底部居中浮层，2.6 秒自动消失

## 第七章 总结与展望

### 7.1 工作总结

本项目基于 Spring Boot 3.5 与 LangChain4j 框架，成功构建了一个集成 RAG 检索增强、MCP 工具调用、SSE 流式对话的智能助手「小智」。系统实现了完整的 AI 对话能力，支持私有知识库问答和联网搜索，具备会话持久化、知识库管理、主题切换、安全护栏等实用功能。同时集成了 Micrometer + Prometheus 监控体系，可追踪每次 AI 调用的 Token 消耗与响应延迟。

### 7.2 不足与展望

1. **多用户隔离**：当前为单用户设计，后续可引入用户系统，实现多用户会话和知识库隔离
2. **安全护栏**：目前仅过滤了两个字，可扩展敏感词库或接入内容安全 API
3. **多模态**：仅支持文本，可扩展图片理解、语音交互
4. **部署优化**：可编写 Docker Compose 实现一键部署（Redis + PGVector + 应用）
5. **对话导出**：支持导出对话记录为 PDF/Markdown
